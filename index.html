concurrent/volatile

一段代码引起思考

执行结果：while循环不会退出,加入volatile关键字之后，程序主动退出，这说明什么？

再多线程，共享变量，其中一个线程进行修改，其他线程不能及时获取到变更后的值，这就是所谓的可见性,加入关键字之后，会立刻获取结果，说明，volatile 能够保证多线程共享变量的可见性

volatile

保证了可见性，并不能保证原子性，接下来，我们分析如何保证的

硬件层面

CPU、内存、磁盘，一般情况，磁盘是拖后腿的

  • CPU高速缓存
  • 增加进程、线程，通过CPU时间片切换提升CPU的使用率
  • 编译器的指令优化，更合理的去利用好CPU的高速缓存 然后每一种优化，都会带来相应的问题，而这些问题也是 导致线程安全性问题的根源。为了了解前面提到的可见性 问题的本质，我们有必要去了解这些优化的过程

CPU高速缓存

高速缓存分为三部分：L1,L2,L3,其中L1,L2是存储是CPU层面，L3是物理内存与CPU直接的

此时，CPU高速缓存确实处理了与主内存速度的问题，但是同时也带来了，缓存一致性问题。

i++，最终结果是<=2;

缓存一致性问题

各自CPU都从自己缓存读取，当主内存刷新时候，自己的CPU内存并没有刷新，产生缓存一致性问题

解决办法：

  • 总线锁

    多核CPU,一个CPU对共享内存进行操作的时候，在总线会LOCK#信息，其他CPU无法通过总线来访问共享内存的数据，这种并不能达到并行计算效果

  • 缓存锁

缓存协议MESI，modify、exclusive，share,invilate

 M：被修改的。处于这一状态的数据，只在本CPU中有缓存数据，而其他CPU中没有。同时其状态相对于内存中的值来说，是已经被修改的，且没有更新到内存中。
E：独占的。处于这一状态的数据，只有在本CPU中有缓存，且其数据没有修改，即与内存中一致。
S：共享的。处于这一状态的数据在多个CPU中都有缓存，且与内存一致。

​ I：无效的。本CPU中的这份缓存已经无效。

*在 MESI 协议中，每个缓存的缓存控制器不仅知道自己的 读写操作，而且也监听(snoop)其它Cache的读写操作 *

# Spring Boot 配置加载

背景

​ 近期启动一个启动，项目结构是多个模块，其中一个是通用模块：Cache，cache模块自身配置一个application.yml,其他模块依赖此模块，并且有自己的application.yml文件

需求

当其他模块依赖cache模块时，突然发现cache模块的配置文件无法加载，why?

试错

根据Spring boot 启动相关事件，定位ConfigFileApplicationListener

官方文档，提供了相关配置项，我们通过官方文档，能发现，有个叫spring.config.name的配置，文档并没有说明多个文件名称如何配置，根据常规多数是通过“,”根据分割，不妨尝试

https://docs.spring.io/spring-boot/docs/2.2.6.RELEASE/reference/html/appendix-application-properties.html#common-application-properties

猜测一：文件名 以 ,进行分割

经过尝试发现失败，spring boot 默认读取的配置文件就是：application，假如你想更换非命名为application的配置文件配置了spring.config.name，无法加载，因为这个文件此时没有被加载

猜测二：既然非application的无法加载，能否通过环境变量配置

通过启动命令 ** java -jar -Dspring.config.name=111.yml,application.yml **
